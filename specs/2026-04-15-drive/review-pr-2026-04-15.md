## Findings

### [WARN] 1. Guided mode missing review-spec gate
**Category:** test-gap
**Description:** The spec's ac-03 says guided mode "pauses at review-spec and review-pr." The SKILL.md §6 pauses at review-pr verdict, but there is no review-spec gate in guided mode. The autonomy table says guided "pauses at review-spec and review-pr" but the stage instructions (§3-4) have no guided-mode AskUserQuestion between design and spec, or between spec and implement.
**Evidence:** SKILL.md line 22: "pauses at review-spec and review-pr for author go/no-go." But §3 (Design) and §4 (Spec) have no guided-mode gates — only supervised has gates at §4 and §5.
**Recommendation:** Add a guided-mode gate after the spec stage (§4), presenting the spec for review before proceeding to implement. This is what distinguishes guided from full — the author gets to review the plan before code is written.

### [WARN] 2. Multiple concurrent drives unhandled
**Category:** edge-case
**Description:** session-context.sh uses `find specs -maxdepth 2 -name 'drive.yaml' | head -1` — if two cards are being driven simultaneously, only the first found is surfaced. The `head -1` is non-deterministic (depends on filesystem order).
**Evidence:** Test 2 confirmed only one drive is shown. The script doesn't warn about the second.
**Recommendation:** Either (a) surface all active drives, or (b) add a comment documenting that single-drive is intentional and the constraint is "one drive at a time." Given this is v1, option (b) is fine — document the limitation.

### [WARN] 3. Budget check off-by-one ambiguity
**Category:** edge-case
**Description:** §7 says "If `iteration > budget` (i.e., this was the 3rd NO-GO), jump to §8." But after 3 NO-GOs, iteration would be incremented to 4 before the check. The parenthetical clarification is correct, but the logic description could be read as allowing iteration 4 to start before checking.
**Evidence:** SKILL.md line 167: step 2 increments iteration, step 3 checks `iteration > budget`. With budget=3, after 3rd NO-GO iteration becomes 4, and 4 > 3 triggers escalation. This is correct but the ordering (increment then check) means the agent must not start design before checking.
**Recommendation:** Reorder: check budget BEFORE incrementing, or clarify: "If this was the last iteration (iteration == budget before increment), jump to §8."
