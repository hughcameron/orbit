## Verdict

**REQUEST_CHANGES**

The SKILL.md is well-structured and addresses most ACs clearly. However, the hook script has a reliably-reproducible bug that emits a shell error on the most common rally state (fresh rally with no complete/parked cards), and several core mechanisms in the SKILL.md are underspecified or internally contradictory in ways that will block a fresh agent from executing them correctly. Most notably: (1) the design sub-agent "tools allow-list" mechanism described in §4a is not how Claude Code's Agent tool works — writes will not be blocked as ac-06 requires; (2) parallel implementation (§7) calls for sub-agents to "follow drive's stage logic inline" including drive's interactive `AskUserQuestion` gates, which Agent-tool sub-agents cannot present; (3) concurrent rally.yaml writes from sub-agents in separate worktrees are not coordinated.

Not BLOCK — the skeleton is sound and these are addressable with focused edits, not a redesign.

## Findings

### F-01 — Hook emits shell error on fresh rally (CRITICAL)

**Category:** Bug — runtime failure
**File:** plugins/orb/scripts/session-context.sh:50-55, 78

**Description:** The counter extraction uses `grep -c PATTERN FILE 2>/dev/null || echo 0`. When `grep -c` finds zero matches, it writes `"0\n"` to stdout AND exits 1. The `|| echo 0` then appends ANOTHER `"0\n"`. The captured variable ends up as the literal string `"0\n0"`. This is then interpolated into a numeric comparison `[[ "$parked_cards" -gt 0 ]]` (line 78) which fails with a shell syntax error. It is also printed into the banner line (`"0\n0 complete, 0\n0 parked"`).

**Evidence:**
```
$ cat /tmp/rally-test-fresh/specs/rally.yaml   # fresh rally, all cards status: proposed
...
$ bash plugins/orb/scripts/session-context.sh
orbit: Active rally — "fresh rally" (guided mode, phase: approved)
  Cards: 2 total, 0
0 complete, 0
0 parked
    - cards/0020-alpha.yaml [proposed]
    - cards/0021-beta.yaml [proposed]
session-context.sh: line 78: [[: 0
0: syntax error in expression (error token is "0")
  Next: run /orb:rally to resume at phase "approved"
```

**Why this matters:** This fires on the most common rally state — immediately after `/orb:rally` is approved (zero complete, zero parked). Every user's first rally session will see a visibly garbled banner and a shell syntax error. This directly violates ac-22's "surfaces rally state at session start" and the evaluation principle of artefact integrity.

**Recommendation:** Replace `grep -c ... || echo 0` with `$(grep -c ... 2>/dev/null || true)` after initialising the variable to zero, or use `wc -l` on a grep pipeline, or use awk/shell counting. Example:
```bash
complete_cards=$(awk '/^    status: complete/ {c++} END {print c+0}' "$rally_file")
```

### F-02 — Hook aborts silently on rally.yaml missing any grep'd field (HIGH)

**Category:** Robustness — silent failure mode
**File:** plugins/orb/scripts/session-context.sh:42-44

**Description:** Under `set -euo pipefail`, if `grep '^rally:'` finds no match (e.g. `rally:` field missing because user corrupted the file or an old format), the `grep | sed` pipeline fails and `set -e` aborts the entire hook with exit 1. No error message, no rally context, no fallthrough to the drive block or the latest-spec block — the user sees nothing.

**Evidence:**
```
$ cat /tmp/rally-test-corrupt/specs/rally.yaml
autonomy: guided
phase: implementing
cards: []

$ bash plugins/orb/scripts/session-context.sh
===EXIT: 1===   # no output
```
Same behaviour when `phase:` is absent. The SKILL.md §12 validation rules say "malformed YAML → clear error naming the line" — the hook does not do this; it produces the opposite (silent death).

**Why this matters:** ac-04 demands actionable errors from rally.yaml validation. ac-22 demands the hook surface rally state. When a hook sees a corrupt file it should either (a) emit a clear warning OR (b) fall through gracefully to the rest of the context detection. Currently it does neither.

**Recommendation:** Either (a) stop relying on `set -e` for the rally block — wrap `rally_goal`, `rally_phase`, `rally_autonomy` reads in `|| true` so empty values are handled downstream, OR (b) check file sanity explicitly first and emit an "orbit: rally.yaml malformed — inspect specs/rally.yaml" warning to stderr before skipping the block. Keep in mind the rest of the script is a best-effort context hook; aborting it silently is a regression relative to pre-rally behaviour.

### F-03 — ac-06 mechanism (sub-agent path constraint) is not implementable as described (HIGH)

**Category:** Spec violation — aspirational mechanism
**File:** plugins/orb/skills/rally/SKILL.md §4a, lines 156-157 and 151

**Description:** The SKILL says "Constrain sub-agent writes via the tools allow-list. Configure each sub-agent's tools parameter so Write/Edit is scoped to the card's `spec_dir`. Claude Code's tools allow-list is the native mechanism — do not add a separate hook."

Claude Code's Agent tool `tools` parameter is a list of tool *names* (e.g. `["Read", "Edit", "Bash"]`) — it does NOT accept per-tool path scopes. There is no native mechanism to say "Write is allowed only to spec_dir/". The only path-scoping mechanisms in Claude Code are PreToolUse hooks (which the spec review said was resolved away) or the system prompt telling the agent where to write (plus trust).

The sub-agent brief (lines 150-153) does instruct "Write your decision pack to: <spec_dir>/decisions.md. Do not write anywhere else. Do not read files outside cards/, specs/, and the source tree." — but that is a prompt-level request, not a blocking constraint. ac-06 verification says "Attempt a write outside its assigned spec directory. Verify the write is blocked." That cannot be verified against the mechanism as prescribed — a non-compliant sub-agent's write would not be blocked.

**Why this matters:** This was the HIGH finding from the spec review that v2.0 claims to have resolved ("HIGH: PreToolUse hook resolved — tools allow-list for design sub-agents"). The v2.0 resolution replaces one non-mechanism (hook) with another (allow-list as described), neither of which blocks writes. ac-06 is therefore not addressed by the implementation.

**Recommendation:** Choose one of three paths: (a) reinstate a PreToolUse hook scoped to the rally's active spec_dirs and document it in the SKILL; (b) downgrade ac-06 to a convention/trust statement — "sub-agents are prompted to write only to spec_dir; violation is treated as sub-agent failure at post-hoc verification" — and update the AC text to reflect that; (c) provide the sub-agent with only a Bash sub-tool restricted via `tools: ["Read", "Grep", "Glob", "Bash(echo:*, ..., tee:spec_dir/*)"]`-style shell-level scoping (if supported). Option (b) is honest; option (a) is robust; the current text is neither.

### F-04 — Disjointness extraction approach unspecified (MEDIUM)

**Category:** Underspecified mechanism
**File:** plugins/orb/skills/rally/SKILL.md §6c, lines 210-216

**Description:** "Extract from each interview: Files named in the design (e.g. `plugins/orb/skills/foo/SKILL.md`); Symbols named (types, traits, functions, schemas); Shared references (skills, scripts, hooks). Compute the intersection."

The extraction algorithm is not specified. interview.md is free-form markdown prose plus Q&A. "Extract files and symbols" could mean: regex on paths and CamelCase identifiers, LLM-driven extraction, structured "Artefacts" sections in interview.md, or something else. A fresh agent reading this cold cannot produce a deterministic extraction without inventing one. ac-10 verification depends on "the definitive check identifies the shared trait" — which implies deterministic extraction.

**Recommendation:** Either (a) require interview.md to include an explicit machine-readable "Surfaces" section (files[], symbols[]) that sub-agents must fill in — this also upgrades ac-10 from prose-heuristic to genuinely deterministic — or (b) prescribe the regex/extraction heuristics in the SKILL (e.g. "grep for paths matching `(plugins|specs|cards|src)/\S+` and for PascalCase/camelCase identifiers in `trait|struct|class|fn|def` contexts"). Either makes ac-10's "definitive" claim real.

### F-05 — Parallel implementation + drive's interactive gates are incompatible (HIGH)

**Category:** Design contradiction
**File:** plugins/orb/skills/rally/SKILL.md §7 Parallel Implementation, lines 260-269 + §7 "follow drive's full stage logic"

**Description:** Parallel implementation launches Agent-tool sub-agents. Sub-agents in Claude Code run headless — they CANNOT present `AskUserQuestion` gates to the author. Yet rally instructs each sub-agent to "Follow drive's stage logic (spec → review-spec → implement → review-pr) inline." Drive's guided mode has an AskUserQuestion gate in §7 (PR-review summary, lines 164-175 of drive SKILL.md): "AskUserQuestion: 'Drive summary for <card name>…'". Supervised mode has more such gates.

The rally SKILL does not explain how a sub-agent satisfies this. Options:
- Sub-agent asks the question → **does not work**; sub-agents have no user channel
- Sub-agent uses `full` autonomy drive → drive's full mode requires ≥3 card scenarios (drive SKILL §1); rally does not verify this, so sub-agent would `REFUSE` at pre-flight on any thin card
- Lead agent presents the final review-pr gate on behalf of sub-agents — plausible, but the SKILL doesn't say this

Rally's §7 parallel flow does say "Each sub-agent runs drive's stage logic within its worktree independently" and "on completion, each sub-agent produces its own PR" — implying full autonomy — but rally's Autonomy Levels table (line 28) says "`full` autonomy is **not offered**". Contradiction.

**Recommendation:** Pick and document one strategy:
- (a) Parallel mode runs sub-agents at full autonomy for implementation + review-pr, and rally verifies all cards have ≥3 scenarios before launching. The final gate is the batched PR review at §8.
- (b) Sub-agents produce artefacts (spec, review-spec, progress, review-pr draft) and return; the lead agent then presents the batched PR review as the single gate. Drive's "guided" AskUserQuestion gates are not invoked inside sub-agents.

Option (b) is cleaner and matches the "rally packs human work into two gates" framing.

### F-06 — Parallel sub-agents writing to rally.yaml across worktrees (MEDIUM)

**Category:** State coordination — race/lost-update
**File:** plugins/orb/skills/rally/SKILL.md §7 Parallel + §9 NO-GO Handling

**Description:** Each parallel sub-agent runs in its own git worktree at `../<repo>-rally-<slug>/`. Worktrees share `.git` but each has its own working tree for untracked files. `specs/rally.yaml` is untracked orchestration state (it's not committed per rally). So each sub-agent's worktree has its own `specs/rally.yaml` copy (or none, if rally.yaml was never committed). When §9 says "rally.yaml records the card's status as 'parked'", which worktree's copy does the sub-agent write to? If the lead is responsible, how does the lead know when a sub-agent status-changes without polling?

The SKILL does not specify: (a) where rally.yaml physically lives relative to worktrees, (b) which agent writes it, (c) how concurrent state updates from N sub-agents are serialised.

**Recommendation:** State explicitly that rally.yaml is owned by the lead agent only and lives in the main worktree. Sub-agents report status by returning a structured message; the lead applies the rally.yaml update. Equivalently, sub-agents can write a `<spec_dir>/result.yaml` status file in their own worktree's filesystem, but the lead merges these into rally.yaml at its cadence. The TaskCreate/TaskUpdate path already implies something like this — the SKILL just needs to say so.

### F-07 — Stacked-PR gap handling: first-card-parked and all-parked cases not illustrated (MEDIUM)

**Category:** Edge case coverage
**File:** plugins/orb/skills/rally/SKILL.md §8 Stacked PRs, lines 286-288

**Description:** §8 says "If a middle card is parked, subsequent PRs target the last non-parked card's branch." The middle case is illustrated. Two missing cases:
- **First card parked** — what does the (new) first non-parked card target? main. Should be obvious, but the recipe says "previous non-parked card's branch (or main for the first card)" (§7 step 2) which works out, but the §8 example only shows the middle case.
- **All cards parked** — no PRs created; rally completion should report zero PRs with all constraints. §10 template handles it structurally (Completed: 0 card(s), Parked: N card(s)), but the rally should not attempt a stack of zero.
- **Last card parked** — stack still valid (just shorter); no wrinkle, but worth one-line acknowledgment.

ac-16 verification only exercises the middle case, so the AC is formally covered; the SKILL could be marginally clearer.

**Recommendation:** Add a two-line note after the §8 stack diagram: "If the first card is parked, the next non-parked card targets main. If all cards are parked, no stack is produced — the completion summary lists constraints only."

### F-09 — awk parser assumes YAML field order (path first per card) (LOW)

**Category:** Fragility
**File:** plugins/orb/scripts/session-context.sh:58-75 and 80-91

**Description:** The awk parser for per-card status and parked constraints assumes each card starts with `^  - path:`. YAML is order-insensitive, so a rally.yaml with a differently-ordered card map breaks the parser (card count drops to 0 in my simulation). The SKILL.md writes cards in the expected order, so this is only triggered by hand-edits or future code changes — but the brittleness violates the principle of "validate on read". MEDIUM only because the SKILL owns writes.

**Recommendation:** Either (a) tolerate any field order by tracking state only within block-indented regions (`^  - ` as start, `^  -` or `^[a-z]` as end), or (b) add a note in SKILL.md that rally.yaml must be written with `path` as the first field of each card map. Probably (a).

## Honest Assessment

The rally SKILL.md is a strong piece of workflow design — the consolidated-decision-pack model genuinely packs human attention onto high-signal gates, the two-layer state (rally.yaml coordination + per-card drive.yaml sub-stage) is principled, and the completion-then-archive lifecycle avoids the common mistake of deleting state eagerly. Most ACs are clearly covered by identifiable sections, and the Critical Rules block reads like a usable operational checklist.

The implementation's weakness is at the seam between what the SKILL *prescribes* and what the runtime can actually *enforce*. Three load-bearing claims — the tools allow-list for design sub-agents (F-03), drive's interactive gates running inside parallel sub-agents (F-05), and coordinated rally.yaml writes across worktrees (F-06) — are stated as if they were mechanical guarantees when they are in fact prompt-level hopes. These need either honest downgrading to "conventions verified at handoff" or concrete mechanisms added. The hook script bugs (F-01, F-02) are straightforward to fix but reliably fire on common states. Nothing here is a redesign-level problem; this is REQUEST_CHANGES, not BLOCK.
