## Findings

### [MEDIUM] Card format divergence — missing `priority` field
**Category:** edge-case
**Description:** The `/orb:card` YAML template includes a `priority` field (`now / next / later`). The distill card template omits it entirely. A distilled card will be structurally distinguishable from an interview-created card by the absence of `priority`. This may cause downstream tooling or humans to treat distilled cards differently.
**Evidence:** `/orb:card` SKILL.md line 60: `priority: "now"` — present in template. `/orb:distill` SKILL.md lines 64-85: no `priority` field in template.
**Recommendation:** Either add `priority` to the distill template (defaulting to omitting it or setting a default like `"next"`) or explicitly document that distilled cards omit priority because it requires human judgment. The spec constraint says "standard card YAML format (feature, as_a, i_want, so_that, scenarios, references)" — priority is not listed as required, so this is arguably compliant. But a note in the integration section would prevent confusion.

### [LOW] `references` field semantic difference
**Category:** edge-case
**Description:** In `/orb:card`, `references` contains prior art descriptions (e.g. `"uv: fast, minimal output"`). In `/orb:distill`, `references` contains file paths (e.g. the source document path). Both are string lists, so they're structurally compatible, but semantically they serve different purposes. A card could end up with a mix if edited later.
**Evidence:** `/orb:card` SKILL.md line 68: `"<tool/approach>: <what's relevant about it>"`. `/orb:distill` SKILL.md line 84: `"<path to source file>"`.
**Recommendation:** This is fine as-is. The spec explicitly requires source path in references, and the formats are compatible. No change needed, but worth noting for awareness.

### [LOW] No instruction for missing `orbit/cards/` directory
**Category:** edge-case
**Description:** Step 3 says "Read the `orbit/cards/` directory" and step 6 says "read `orbit/cards/` directory at write time." If `orbit/cards/` does not exist (e.g., brand new repo before any `/orb:card` or `/orb:init` has run), the behavior is undefined. The agent might error or might create the directory — neither is specified.
**Evidence:** SKILL.md step 3 (line 53-56) and step 6 (line 115) both assume `orbit/cards/` exists.
**Recommendation:** Add a brief note: "If `orbit/cards/` does not exist, create it. Start numbering at 0001." This mirrors `/orb:card` step 1 which says "If no cards exist, start at `0001`" — but `/orb:card` also doesn't explicitly handle directory creation, so this is a shared gap.

### [LOW] Edit round adds scenarios without source — potential constraint violation
**Category:** edge-case
**Description:** Step 5 allows the user to request edits like "add a scenario for error handling." If the user requests a new scenario during editing, the agent must fabricate `source_lines` for content that may not exist in the source document. This creates tension with the "extract, don't invent" rule and the mandatory `source_lines` constraint.
**Evidence:** Step 5 example (line 106): "change the feature name to X and add a scenario for error handling". Step 4 critical rule (line 89-90): "Every scenario MUST trace to something in the source material."
**Recommendation:** Add guidance for this case. Options: (a) user-added scenarios during edit are exempt from `source_lines` (set to `"user-requested"` or similar), (b) the agent should ask the user to point to a passage, or (c) the agent should note that user-requested additions that don't trace to source will omit `source_lines`. This is a genuine ambiguity that an agent will encounter.

## Honest Assessment

This is a clean, well-structured SKILL.md that faithfully implements all 10 acceptance criteria from the spec. The instructions are clear enough that an LLM agent could follow them without ambiguity in the happy path. The one finding worth addressing before merge is the tension between the edit flow (step 5) and the mandatory `source_lines` constraint — a user asking "add a scenario for error handling" during edit will put the agent in an impossible position where it must either invent source attribution or refuse a reasonable edit request. Adding a single sentence of guidance resolves this. The missing `priority` field is a minor divergence from `/orb:card` format that should at minimum be acknowledged. Overall, this is close to merge-ready with one targeted fix needed.
