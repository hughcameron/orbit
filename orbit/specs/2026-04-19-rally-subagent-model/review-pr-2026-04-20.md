## Findings

### [LOW] ac-17 window tuning is vacuous against the current SKILL.md
**Category:** honesty-violation (mild)
**Description:** Spec ac-17 mandates "the implementation MUST first run the scan against the current rally SKILL.md and tune the window down to the smallest value at which every legitimate existing claim still passes, recording the tuned value in the script with a comment citing this AC." The script records `WINDOW_CHARS=80` with a comment saying it was tuned. Because the current SKILL.md contains **zero** enforcement verbs in §2/§4/§7/§11 (grep-verified against the verb regex), no legitimate claim exists to tune against — any window value from 1 to infinity would produce the same (empty) pass set. The tuning comment at `rally-coherence-scan.sh:84-87` states the 80-char default passes cleanly, which is technically true but trivially so.
**Evidence:** `Grep` for `\b(enforced|enforces|blocked|blocks|prevented|prevents|guaranteed|guarantees)\b` (case-insensitive) over SKILL.md returns no matches; the scan's own output includes the honest line `(no enforcement verbs in named sections — trivially passes)` in progress.md (though SKILL.md-on-disk now omits this explanatory line, replaced by a blank line under `== Adjacency scan (window=80) ==`).
**Recommendation:** Either (a) add a comment to the script clarifying "no existing claims to tune against; 80 retained as default" to preserve honesty, or (b) restore the explanatory "no enforcement verbs in named sections — trivially passes" line to the scan output so human readers understand a blank adjacency block is the expected state, not a silent bug. Non-blocking; a follow-up cleanup.

**Recommendation:** Update progress.md evidence block to match actual output, or add the explanatory echo to the script. Cosmetic.
