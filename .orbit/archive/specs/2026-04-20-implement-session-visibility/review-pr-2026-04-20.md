## Findings

### [LOW] Closed gate-AC produces silent empty post-gate-ac output
**Category:** edge-case
**Description:** `parse-progress.sh post-gate-ac <file> <gate-id>` iterates only over `- [ ] ac-NN` lines (unchecked), so when the named gate AC is already `[x]`, the walk never matches the gate id and the function emits nothing. This is fine for the current SessionStart caller (it only calls post-gate-ac when `next_is_gate==1`, i.e. an unchecked gate), so there is no user-visible bug today. Logging this as a latent contract issue: the subcommand's header docs imply "first unchecked AC declared after the named gate", which would still be meaningful if the gate is closed. The implementation silently returns empty in that case.
**Evidence:** parse-progress.sh:157–165 (awk filter `/^- \[ \] ac-[0-9]+/`). Reproduced with `/tmp/isv-fixtures/progress.md` where ac-01 is `[x]`; `post-gate-ac ... ac-01` returns empty while ac-02 is unchecked.
**Recommendation:** None for this merge — the single caller's precondition (unchecked gate) is enforced upstream. If a future caller needs "after any gate, open or closed", broaden the awk filter to match `- \[[ x]\] ac-[0-9]+` for the gate-detection step while still filtering unchecked for the emission.
