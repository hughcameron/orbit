No external reference in the repo points to "§9 step N" by number, so the renumbering is self-contained. No change recommended.

### [LOW] Verbatim prompt body is inline rather than in a template file

**Category:** maintainability
**Description:** The heartbeat prompt body is inlined in SKILL.md rather than stored in a template file adjacent to the skill. Given the body is short and tightly coupled to the surrounding specification, inlining is appropriate — extracting to a template would add indirection without benefit. Future refactor could revisit if the body grows or is reused elsewhere.

## Environment Simulation

The implementation is a documentation change to a Claude Code skill definition — there is no runtime code path to simulate. The cron-tool primitives (CronCreate, CronList, CronDelete) are Claude Code harness tools whose runtime behaviour lives outside this repo. The SKILL.md changes define the contract the harness follows when drive invokes these tools; the contract will be exercised when a live `/orb:drive .orbit/cards/0005-drive.yaml full` runs against a card that has at least one such drive attempt.

No evidence contradicts the design assumptions (task restoration on `--resume`, session-scoped cron tasks, synchronous `CronCreate`/`CronList`/`CronDelete` semantics) — all four are drawn from the interview's cited tools-reference source lines on the card.

## Edge Case Probing

- **First run:** §2a gate is `autonomy == full`; guided/supervised drives skip heartbeat scheduling entirely. No spurious cron creation on the common non-full paths.
- **CronCreate failure (harness does not support cron):** ac-13 explicitly covers this — drive logs `heartbeat unavailable: <reason>` and continues. The pipeline is not gated on the heartbeat.
- **CronDelete failure (task already expired):** ac-05 covers both the §9 and §10 paths with `heartbeat cleanup skipped: <reason>` and explicit non-fatal semantics.
- **Resume after cron task expired:** §11 step 6 re-runs the idempotent reconciliation — CronList returns no match, CronCreate re-establishes the heartbeat. Covered.
- **Resume after cron task restored:** §11 step 6 — CronList finds the task, no-op. Drive never delete-then-recreates (Critical Rules bullet).
- **Heartbeat fires mid-forked-review:** prompt body read-only contract + Critical Rules bullet. The heartbeat cannot mutate state or spawn work during a fork. Covered.
- **Heartbeat during 30-second escalation window:** §9 step 3 runs CronDelete *before* step 4 schedules the ping — the recurring heartbeat cannot fire in the window between them.
- **`request changes` at MEDIUM+ four-option prompt:** §7a says drive increments `review_cycles.<stage>` and re-enters the review cycle, budget-gated per §5a. This is consistent with how a forked REQUEST_CHANGES verdict would be handled. The author is responsible for modifying the spec/implementation to address the findings; drive does not pass freeform text to the re-review.
- **`read full review first` deferral with no follow-up:** §7a says "The deferral is open-ended; drive does not time out." Drive will wait indefinitely; the re-prompt fires on the author's next turn whenever that comes.

## Honest Assessment

This is a clean, surgical extension to `/orb:drive`. Every AC maps to a concrete edit in SKILL.md, and every constraint is carried through into the text. The scope discipline is good — nothing outside `plugins/orb/skills/drive/SKILL.md`, `.orbit/cards/0005-drive.yaml`, and the spec dir is touched, which is precisely what the card and spec required.

The three biggest risks — heartbeat safety during forked reviews, cron task survival across `--resume`, and the MEDIUM+ prompt not accidentally firing on REQUEST_CHANGES/BLOCK — are all explicitly addressed in the text with load-bearing language (read-only prompt body, idempotent CronList-first, APPROVE-gate-only scope). The renumbering of §9 introduces minor documentation churn but no external references break.

Ready to ship.
