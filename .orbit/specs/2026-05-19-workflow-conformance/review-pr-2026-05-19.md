## Findings

### [LOW] ac-05 pin-bump integration test not implemented
**Category:** test-gap
**Description:** The ac-05 verification text calls for an "Integration test: invoke `orbit setup --bump-pin` ... against a fixture with `plugin_version: \"0.4.18\"` and the binary at 0.4.20; assert post-invocation config has `plugin_version: \"0.4.20\"`." No such test exists. The remediation verb string `"orbit setup --bump-pin"` is hard-coded as the finding handle, but no orbit-state CLI implements `setup` (it's a plugin-side `/orb:setup` skill), and no documentation of the equivalence exists in `plugins/orb/skills/setup/SKILL.md`.
**Evidence:** `grep -rn "bump-pin\|bump_pin"` in the orbit-state tree shows only the literal-string assertion in the unit test and the finding builder — no integration test, no skill-side documentation. The spec ac-05 itself permits the latter: "implementing agent adds the flag if absent, OR documents that `orbit setup` already rewrites the pin and `--bump-pin` is an alias." The implementer did neither.
**Recommendation:** Either (a) add a paragraph to `plugins/orb/skills/setup/SKILL.md` documenting how the `/orb:setup` skill rewrites `plugin_version` on re-run (which closes the documentation half of ac-05's flexibility clause); or (b) defer to a follow-on spec carrying the implementation-side pin-bump. Not a verdict-blocker — the dominant finding still ships, escalates correctly, and the agent gets a parseable handle. The integration-test omission is a documented test-gap rather than a behavioural bug.
