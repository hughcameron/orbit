## Findings

### [HIGH] AC-09's "expected target path" pivot is undefined and contradicts the auto-generated-name acceptance

**Category:** missing-requirement
**Pass:** 2

**Description:** AC-09 closes "the chain-detection mechanism consults the curator state — if a chain's expected SKILL.md target path is already an archived routine, the agent does not re-author for the same chain in subsequent sessions." This presupposes a deterministic function `chain → expected target path` that is stable across sessions and after author edits.

The spec does not pin that function. AC-03 says the agent writes to `.claude/skills/<auto-generated-name>/SKILL.md` and refers to an "auto-generated-name" without naming the algorithm. The tabletop trade-off table (`tabletop.md:34`) accepts "Auto-generated skill names — Agent picks a name — acceptable — author renames" as a v1 simplification.

Combined, these produce a contradiction:

1. **Author rename case.** Agent writes `.claude/skills/foo-chain/`. Author renames to `.claude/skills/release-loop/`. Next session, agent re-detects the same chain → computes `.claude/skills/foo-chain/` → not archived → writes a second routine. The "do not re-author" branch of AC-09 silently fires the wrong way, producing duplicates.
2. **Naming non-determinism.** If the agent's name generation depends on session context (model temperature, time-of-day cues, varying chain shape after the ≤1-skipped-step allowance — see Finding 3), the same chain may compute different paths across sessions, defeating AC-09's archive-state lookup.

This is also untestable as written: AC-09's fixture says "author archives a routine and the agent's next detection run on the same chain produces no write" — but the test cannot pass without the name-generation algorithm being deterministic and pinned.

**Evidence:**
- `spec.yaml` ac-03 — `<auto-generated-name>` (no algorithm)
- `spec.yaml` ac-09 — pivots on "expected SKILL.md target path"
- `tabletop.md:34` — accepts agent-picks-a-name with "author renames" as a v1 simplification
- v1 review Finding 3 raised the rejection-substrate question; v2 moved the lookup to "expected target path" but did not pin the path function

**Recommendation:** Pin one of these in the spec:

1. Add an AC requiring chain identity to be a deterministic function of the ordered skill_id sequence (e.g. a chain-hash captured in the routine SKILL.md's front-matter as `chain_id: <hex>`), AND store the archive lookup as a content-based check (chain_id present anywhere under `.claude/skills/.archive/`), not a path-based check.
2. Or accept the duplicates-on-rename failure mode explicitly and reframe AC-09 to "the agent does not re-author until the author archives the existing routine (irrespective of its name)" — which means the lookup is by chain identity, not path, anyway.

Either resolves the contradiction. Leaving AC-09 path-based while ACs above accept "agent picks a name, author renames" guarantees the test fixture either over-specifies the agent or under-specifies the substrate.

### [MEDIUM] AC-06 makes `orbit audit conformance` a mutating verb without naming the invariants

**Category:** failure-mode
**Pass:** 2

**Description:** AC-06: "the audit run that confirms every `/orb:<verb>` reference in a routine's SKILL.md body still resolves to a live skill updates the routine's `last_verified` field to the audit's timestamp; … subsequent updates happen on each passing audit."

This adds a write side-effect to `audit conformance` — today a read-only aggregator (`AuditConformanceResult` in `orbit-state/crates/core/src/verbs.rs:1103`). Mutating audit raises invariants the spec doesn't pin:

- **Read-only-ness contract.** Existing conformance subsystems (`cards | memos | setup`) emit findings without touching the artefacts they describe. AC-06 changes that for the routine subsystem — invariant break unless explicitly carved out.
- **Front-matter editing.** Writing `last_verified` means parsing the SKILL.md, mutating front-matter, and rewriting the file. Atomicity / lock semantics not named. If the audit runs concurrently with the agent re-authoring or the author editing, race conditions are possible.
- **CI vs interactive runs.** If `audit conformance` runs in CI (read-only context), the write must either skip or fail loudly. The spec says "subsequent updates happen on each passing audit" with no environment carve-out.
- **Audit idempotence.** Two passing audits on the same file produce different file content (different timestamps). Any CI invariant that conformance is byte-equal across runs breaks.

The tabletop kill-condition K5 explicitly bets the approach on "audit-driven `last_verified` reliably catches drift modes in practice." Shipping AC-06 without the invariants pinned means the implementing agent must improvise — exactly the failure shape v1 Finding 2 warned about for a *different* mechanism.

**Evidence:**
- `orbit-state/crates/core/src/verbs.rs:1097-1146` — `AuditConformanceResult` and `ConformanceFinding` are read-only result shapes; no write API on this verb today
- `spec.yaml` ac-06 — names the write but no invariants
- `spec.yaml` ac-07 — finding-family addition only (the read side); does not name the write
- `tabletop.md:91-94` (K5) — bets the approach on this mechanism being reliable

**Recommendation:** Pin the write-path invariants on AC-06. Two acceptable shapes:

1. Split the write off audit: introduce a small `orbit routine verify <path>` verb that does the resolve-check, writes `last_verified` on pass, and is *called by* `audit conformance` when running in a write-enabled context; the conformance result reports the verify outcome but doesn't itself write. The split keeps the audit aggregator read-only.
2. Keep the write inside audit but carve out: name a `--write` / `--read-only` flag, default to the safer side (read-only in CI, write-on-pass interactively), and add explicit ACs covering atomicity and concurrent-edit behaviour.

Either resolves the invariant gap. Leaving it implicit means the implementing agent will pick unilaterally and the chosen shape will leak into adjacent audit verbs.

### [MEDIUM] AC-02's "exact-ordering" contradicts the tabletop's "≤1 skipped step" allowance

**Category:** constraint-conflict
**Pass:** 2

**Description:** AC-02: "exact-ordering chains of length two or more across two or more sessions trigger authoring; chains observed only once or with inconsistent ordering are recorded but do not trigger authoring."

Tabletop Implementation Notes (`tabletop.md:102`): "Recurrence detection. Allow ≤1 skipped step in chains of ≥3 for consistency-counting (catches minor variations). Pick the longest consistent chain when ambiguous (avoids re-proposing sub-chains)."

These conflict. "Exact-ordering" excludes the ≤1-skipped-step allowance; "longest consistent chain" requires a longest-prefix or longest-common-subsequence rule that's also not in AC-02. The implementing agent will read AC-02 and write strict-equality matching, then either:

- ship and miss the variations the tabletop wanted caught (false negatives), or
- escalate (round-trip cost) when fixture authoring surfaces the conflict, or
- silently fold the tabletop notes into the implementation without an AC capturing it (drift from spec to substrate).

This is the same shape as v1 Finding 5 — tabletop intent did not survive promotion to the AC. The mechanism is correct in the tabletop and incomplete in the spec.

**Evidence:**
- `spec.yaml` ac-02 — "exact-ordering" + "inconsistent ordering does not trigger"
- `tabletop.md:102` — "≤1 skipped step in chains of ≥3" + "pick the longest consistent chain"

**Recommendation:** Rewrite AC-02 to reflect the tabletop's accepted matching rule. Either:

1. Add the ≤1-skipped-step relaxation and longest-consistent-chain pick rule to AC-02's `when` clause, with the fixture JSONL covering both the relaxation case and the longest-match case.
2. Or move the relaxation out of v1 (accept that v1 is strict exact-matching and the variations land in v2), and update `tabletop.md`'s Implementation Notes to match — the tabletop notes should not commit the spec to a mechanism the ACs don't carry.

Either resolves the conflict. Pinning the matching rule in the AC is the single highest-leverage edit before implementation, because the matcher is the substrate everything downstream tests against.

### [LOW] AC-07 finding-family slug not pinned

**Category:** test-gap
**Pass:** 1

**Description:** AC-07 says `audit conformance` emits a finding family for routines, but does not name the `subsystem` slug. The `ConformanceFinding` schema (`orbit-state/crates/core/src/verbs.rs:1129-1146`) takes a `subsystem: String` field, with current slugs `cards | memos | setup`. Without pinning the new slug, the AC closure shape is "any new family for routines" — testable but the contract is fuzzier than the 2026-05-19-workflow-conformance ACs which name their slugs precisely.

Adjacent ACs in the existing conformance suite name `state` slugs (`stale | byte_drift | missing | pin_behind | pin_ahead`); AC-07 names two state triggers ("last_verified older than 30 days, broken refs") but doesn't propose the state slugs either.

**Evidence:**
- `orbit-state/crates/core/src/verbs.rs:1129-1146` — `ConformanceFinding.subsystem` and `state` are slug strings with conventional values
- `spec.yaml` ac-07 — no subsystem slug, no state slugs

**Recommendation:** Add to AC-07: "subsystem slug `routines`; state slugs `stale` (last_verified > 30 days) and `broken_refs` (one or more references unresolved); remediation.verb is `/orb:audit` or equivalent." A two-line addition; closes the test-shape ambiguity cheaply.

## Honest Assessment

The v2 spec resolves the v1 review cleanly. Substrate adequacy, verification mechanism, author lever, AC descriptions, AC types, and the cross-card commit are all now structurally present. The v2 spec is materially closer to implementable than v1.

The three new findings cluster around one shape: *mechanism is right in the tabletop, not yet sharp enough in the ACs*. AC-09's path lookup contradicts the auto-generated-name acceptance; AC-06's audit-mutates-skill leaks invariants; AC-02 drops the tabletop's matching-rule relaxation. None of these are blocking architectural problems — each is a small, surgical edit to the AC text that the spec author can make in minutes.

The biggest remaining risk is AC-09's path-determinism gap (Finding 1). It's the only finding where the failure mode is silent — the agent will produce duplicate routines on author rename, not surface an escalation. The other two findings will manifest as escalations on first implementation; this one can ship and fail in production. Worth fixing before drive proceeds.
