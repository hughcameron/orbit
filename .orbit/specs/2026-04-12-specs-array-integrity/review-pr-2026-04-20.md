## Findings

### [CRITICAL] Production evidence that ac-01 does not reliably trigger
**Category:** bug / post-ship verification
**Description:** The core guarantee of this spec is that `/orb:spec` appends the new spec path to the card's `specs` array. A post-ship audit of live cards shows this guarantee is already broken:
- `.orbit/cards/0007-drive-forked-reviews.yaml` has `specs: []`
- `.orbit/specs/2026-04-20-drive-forked-reviews/spec.yaml` exists and targets that card
- `.orbit/specs/2026-04-20-drive-forked-reviews/interview.md` line 5 contains `**Card:** .orbit/cards/0007-drive-forked-reviews.yaml` — the exact pattern step-5 of `/orb:spec` claims to parse

**Evidence:**
```
$ grep -A2 "^specs:" .orbit/cards/0007-drive-forked-reviews.yaml
specs: []

$ grep "Card:" .orbit/specs/2026-04-20-drive-forked-reviews/interview.md
**Card:** .orbit/cards/0007-drive-forked-reviews.yaml
```
The spec that shipped *after* this one (2026-04-20) orphaned itself — the very failure this work was commissioned to prevent.

**Recommendation:** Either (a) tighten `/orb:spec` step 5 so the agent is programmatically required to Edit the card YAML before completing (e.g. a concrete "After Writing spec.yaml, Edit the card file — if the interview contains a Card: line, this step is non-optional; failure to Edit is a spec failure") or (b) add a repair step to `/orb:design`'s reconciliation that writes back orphans it surfaces (the current text says "if the author confirms, append the spec path" — this covers only one path). A deterministic hook is an honest alternative given one of the spec's constraints prohibited hooks on write — the prohibition may be what created this failure mode.

### [HIGH] ac-08 constraint violated by design — fallback note not in spec/SKILL.md or design/SKILL.md
**Category:** test-gap / spec-violation
**Description:** AC-08 says: *"Both /orb:spec and /orb:design SKILL.md files mention the grep fallback with rationale"*. Verification: *"Both /orb:spec and /orb:design SKILL.md files mention the grep fallback with rationale"*.

Searching the shipped files:
```
$ rg -i "grep|ripgrep|fallback" plugins/orb/skills/spec/SKILL.md
(no matches)
$ rg -i "grep|ripgrep|fallback" plugins/orb/skills/design/SKILL.md
(no matches)
```

Neither file mentions grep, ripgrep, or fallback. The shared `/orb:keyword-scan` technique does, but `/orb:spec` does not even reference keyword-scan — it was not wired into the spec skill at all. `/orb:design` references keyword-scan but doesn't restate the fallback.

The implementer's progress.md argues "keyword-scan is the shared reference" satisfies ac-08, but that's a delegation pattern the AC's verification text does not endorse.

**Recommendation:** Either (a) rewrite ac-08 (future spec) to say "…reference /orb:keyword-scan, which documents the grep fallback", or (b) add a one-liner to both SKILL.md files noting "if rg unavailable, grep -rl works — see /orb:keyword-scan". Option (b) is cheaper and satisfies the AC as written.

### [MEDIUM] `/orb:spec` step 5 never references `/orb:keyword-scan`
**Category:** consistency
**Description:** The scope expanded from "update the card's specs array" to "wire keyword-scan into 6 skills" (documented in progress.md). `/orb:spec` is one of those skills in spirit — it's the write-time enforcer — but its shipped text has no mention of keyword-scan. If `/orb:spec` is intended to stay lean here, that's fine; but then ac-08's requirement for a fallback note in spec/SKILL.md is doubly unsatisfied.

**Evidence:** plugins/orb/skills/spec/SKILL.md step 5 is self-contained — card read, append, write. No keyword search, no fallback discussion.

**Recommendation:** Make the design decision explicit: is `/orb:spec` a write-time enforcer only (no keyword-scan)? If yes, adjust ac-08 accordingly in a follow-up spec.

### [MEDIUM] `/orb:spec` step 5 silently swallows the "no card" case
**Category:** edge-case
**Description:** Step 5.2 says *"If no card is identified, skip this step — not all specs originate from a card"*. Combined with the "non-negotiable" phrasing two lines later, this creates a loophole: any spec whose interview lacks a `**Card:**` line (or whose interview is missing entirely) quietly bypasses the guarantee. Given ac-02 requires the skill to parse `Card: .orbit/cards/NNNN-slug.yaml`, an interview using a different format (e.g. `Card: 0007` or no prefix) will silently skip.

**Recommendation:** Narrow the skip condition — the agent should at least prompt "no card detected; is this spec orphan by design?" before skipping. Otherwise the "non-negotiable" phrasing doesn't match the behaviour.

### [LOW] Spec lacks `test_prefix` — audit skill cannot scan this spec
**Category:** test-gap
**Description:** Decision `0002-ac-test-prefix.md` introduces `test_prefix` for AC namespace disambiguation. This spec dates 2026-04-12; the test_prefix decision was adopted later in the repo. For prose-only specs, the test_prefix is irrelevant — but `/orb:audit` likely flags this spec as prefix-missing. Note for historical record: acceptable given the ac_types are doc/code-but-prose.

**Recommendation:** None — spec is grandfathered. Future prose-only specs should still declare a test_prefix for auditability, even if tests don't exist.

### [LOW] Progress.md marks ac-08 complete with a delegation argument
**Category:** process
**Description:** The implementer self-reported all 13 items as done, including ac-08 with the comment "keyword-scan is the shared reference". A fresh reviewer reading only the spec and the diff (which is the intended contract of review-pr) would not accept this — the AC's verification text names both SKILL.md files specifically. This is a minor signal that the implementer softened an AC during implementation rather than escalating.

**Recommendation:** Calibration — when ac wording turns out to be too strict during implementation, escalate to the author before marking complete. This is cheaper than a REQUEST_CHANGES cycle post-hoc.

## Honest Assessment

The spec shipped the documentation and shared-technique work competently — the keyword-scan skill is well-factored and got wired into 6 downstream skills, which is genuinely useful scope expansion. On paper the ACs look satisfied. In practice, the most important guarantee (ac-01: write-time enforcement) already failed within 8 days of shipping: card `0007-drive-forked-reviews.yaml` is orphaned from its spec, which is the exact defect this spec was commissioned to prevent. The root cause is that the "enforcement" is a prose instruction in a SKILL.md — agents can and do skip it. That, combined with ac-08's literal-reading violation, makes this a REQUEST_CHANGES rather than APPROVE. The honest verdict is: good design, correct scope, but the non-deterministic enforcement assumption was wrong, and a follow-up spec is needed either to add a hook/script or to harden `/orb:spec`'s step 5 into a gate that the agent cannot finish the skill without satisfying.
