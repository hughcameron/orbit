## Findings

### [LOW] Implementation landed directly on `main`, no separate PR
**Category:** process
**Description:** The spec was opened, reviewed (cycle 1 REQUEST_CHANGES, cycle 2 APPROVE), and then implemented in a single commit (`8f094e9`) authored on `main`. Convention elsewhere in this repo (see `tree-views` recent history) is one commit per AC on a feature branch with a PR. This single-commit landing makes per-AC bisecting harder if a regression later surfaces.
**Evidence:** `git log --oneline` shows `8f094e9 spec: 2026-05-12-reconcile-mode — orbit canonicalise --reconcile` as the only implementation commit; `gh pr list --state all` shows no PR for reconcile-mode.
**Recommendation:** Accept as-is for this spec — the commit message is unusually thorough (43-line body enumerating each LOW review finding's resolution), the test count delta is stated explicitly, and the file list is reviewable. For future specs of this size, prefer the per-AC commit cadence; the conversation here is a process note, not a blocker.

### [LOW] Sidecar grows monotonically; same `path` with differing `value` entries accumulate
**Category:** edge-case
**Description:** `merge_sidecar` treats an entry as duplicate only when both `path` AND `value` match an existing entry. If a canonical file regains a previously-quarantined field with a *different* value, the sidecar gains a second entry at the same `path` with the new value while the old one stays. The spec's ac-06 wording allows this ("existing-entry equality is checked at the parsed-`Value` level"), and the rationale is sound (the substrate never destroys content), but a sidecar with three entries at `path: predecessor_evidence` will surprise a re-anchoring human.
**Evidence:** Synthetic tree at `/tmp/tmp.41flsVQoon/.orbit/specs/0001-test/spec.legacy.yaml` after the Phase 4 third-run probe contains two entries with `path: predecessor_evidence` (values `some top-level legacy prose` and `same prose`).
**Recommendation:** Leave as-is for v1; the behaviour matches the spec literally and the alternative (overwriting on path-match) would be the silent-content-destruction the spec explicitly forbids. Note as a follow-up memo if a real project hits a sidecar with same-path-different-value entries that confuse re-anchoring.

### [INFO] Pre-existing `orbit verify` exit-code wrinkle (not introduced by this spec)
**Category:** environment-mismatch
**Description:** During Phase 3 probing, `orbit verify` on a substrate with a Spec carrying an unknown `version` field prints `drift detected ... parse failed: ... unknown field` but exits 0. Same for `orbit canonicalise` (no `--reconcile`) — `parse_failed: 1` is reported but exit is 0. Both are pre-existing behaviours unrelated to the reconcile-mode change (this spec did not touch `verify_all` or the non-reconcile `run_canonicalise` exit logic).
**Evidence:** Phase 3 transcript: `orbit verify: drift detected ... exit: 0`.
**Recommendation:** Out of scope for this review. Note separately if Hugh wants the routine verify gate to honour parse failures as non-zero — but that is a different spec entirely.

## Honest Assessment

The implementation cleanly satisfies all eight acceptance criteria and ships every framework wire the spec demanded. The reconcile module is well-documented, tightly scoped (the permissive read lives in `reconcile.rs` and nowhere else), and its 13 unit tests plus the three new `FIELDS`-lockstep schema tests cover the meaningful behaviour surface: registry-driven map/drop, default-to-quarantine for unknown content at both top-level and inner-list-of-struct depth, dry-run that doesn't write but signals via exit code, sidecar-first crash-safe write ordering, idempotent re-runs with parsed-`Value` logical equality for merge, and per-entity walking across all four canonical types.

Phase 3 live probing against a synthetic dirty tree confirmed end-to-end: dispositions surface correctly with structural paths (e.g. `acceptance_criteria[1].predecessor_evidence`), the canonical file is rewritten through the canonical writer (deterministic field order, default arrays materialised), the sidecar lands with `{path, value}` entries, a re-run is a no-op, and `orbit verify` then passes against the canonicalised tree — closing the loop the spec describes. The ac-01 strict-parse invariant is intact: routine `orbit verify` and routine `orbit canonicalise` reject unknown fields with the original clear error message, and the only path that accepts them is the explicit `--reconcile` flag.

The absence of `acNN`-prefixed test names is a convention mismatch, not a coverage gap. This crate's 171-test baseline does not use that convention either — every existing test names a behaviour (e.g. `verify_detects_unknown_field`, `spec_version_dropped_per_registry`). Forcing `acNN` prefixes here would have created a precedent inconsistent with the rest of `orbit-state/`. The coverage table above maps each AC to its behavioural test(s) and live-probe evidence; coverage is real.

The single LOW finding worth Hugh's attention is the process note: this spec landed as one commit on `main` rather than a per-AC PR. Accept for this spec given the commit message's unusual thoroughness; nudge future specs of this size toward the per-AC cadence. Verdict: APPROVE.
