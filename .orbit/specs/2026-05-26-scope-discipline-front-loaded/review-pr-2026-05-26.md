## Findings

### [LOW] Duplicated `strip_closed_at` helper across two parity test crates
**Category:** test-gap (minor refactor opportunity)
**Description:** The `strip_closed_at` helper is duplicated verbatim in `orbit-state/crates/cli/tests/parity.rs` and `orbit-state/crates/mcp/tests/parity.rs`. Both crates already share a `common/mod.rs` for envelope fixtures; the helper could live there.
**Evidence:** `git diff main...HEAD -- orbit-state/crates/cli/tests/parity.rs orbit-state/crates/mcp/tests/parity.rs` shows the same 16-line function in both files.
**Recommendation:** Move to either common module so future `closed_at`-format changes need only one edit. Non-blocking — duplication is localised and well-commented.

### [INFO] `strip_closed_at` assumes leading comma in the field's JSON shape
**Category:** edge-case (documented assumption)
**Description:** The pattern `","closed_at":""` relies on `closed_at` never being first in the serialised JSON. Per the struct layout `closed_at` is the LAST field, so the assumption holds today, but a future reorder of `Spec`'s fields would silently break the helper.
**Evidence:** `parity.rs:790` comment says "since `id` comes alphabetically first" — actual ordering is struct-declaration order (`serde_json` derive), not alphabetical, but `closed_at` is last either way.
**Recommendation:** Tighten the comment to reference struct-declaration order rather than alphabetical, OR replace the helper with a JSON parse-mutate-reserialise pass. Non-blocking — the assumption holds and the helper is test-only.

### [INFO] Audit smoke is a vacuous pass on first run
**Category:** observation (called out in spec)
**Description:** AC-08's threshold check returned 0 findings — exactly as AC-05's audit-window gate guarantees. The observation has no signal until specs start closing post-2026-05-26 and accumulating deferrals.
**Evidence:** `orbit audit conformance --json | jq '[.data.result.findings[] | select(.subsystem == "cards" and (.state | startswith("card_coverage")))] | length'` → `0`.
**Recommendation:** No action — the AC itself names this as expected first-run behaviour. The threshold's load-bearing window is future, not present.
