## Findings

### [LOW] No CLI-level test exercises the `spec resolve` clap surface
**Category:** test-gap
**Description:** All 11 new tests live in `orbit-state-core::verbs::tests` and exercise `execute(VerbRequest::SpecResolve(...))` directly. The CLI's `SpecAction::Resolve` parsing path (clap → `build_request` → `VerbRequest::SpecResolve`) is exercised only by manual CLI smoke (which I ran during implement and confirmed: bound-card → `outcome:resolved source:bound_card`; unbound multi → `outcome:prompt source:unbound`; unbound zero-spec → terminal D5 message).
**Evidence:** No `clap`-shaped test in `orbit-state/crates/cli/src/main.rs`; manual smoke output captured in the implementation diff's commit message context.
**Recommendation:** Add an integration test using `assert_cmd` or similar for the CLI binary (defer to a follow-up spec under card 0020-orbit-state — this is an established pattern for the CLI crate rather than a per-verb need).

### [LOW] `goal_first_line` could collapse on multi-byte truncation
**Category:** edge-case
**Description:** `first_line(&s.goal)` returns everything up to the first `\n`. If a spec's goal is a single long line with no newlines, the menu label is the entire goal. The interview open-item flagged this as "implementation detail, picked at coding time" — I picked the simplest shape (newline-bounded) deliberately. Risk is bounded: spec goals are conventionally one sentence; pathological multi-paragraph goals would land as one long line in the menu, which is mildly annoying but not broken.
**Evidence:** `verbs.rs::first_line` returns `s.lines().next().unwrap_or("")` — no character cap.
**Recommendation:** Defer. Add a width cap only if a real menu lands with a goal >~120 chars in a user-visible repro.
