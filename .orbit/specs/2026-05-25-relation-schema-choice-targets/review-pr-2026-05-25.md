## Findings

### [LOW] orbit-state/.orbit/schema-version leaked into the commit — should be gitignored

**Severity:** LOW
**Category:** hygiene
**Description:** Commit `d10825d` added `orbit-state/.orbit/schema-version` (15 bytes, `version: '0.5'`). Per the toplevel `.orbit/.gitignore`, `schema-version` is on the "orbit-state runtime artefacts — local to each clone, never tracked" list, and the orbit-state code's own `layout.rs:12` comment marks it `(substrate-written, gitignored)`. The leak happened because there is no `.gitignore` inside `orbit-state/.orbit/` — when the implementer ran orbit commands from inside the `orbit-state/` crate root, the substrate initialisation wrote the file, and `git add` swept it up.

This is `git log --all --oneline -- orbit-state/.orbit/` returning exactly one commit (this one) — first time anything in `orbit-state/.orbit/` got tracked.

**Evidence:**
- `git ls-files orbit-state/.orbit/` → `orbit-state/.orbit/schema-version` (sole entry)
- `cat .orbit/.gitignore` lists `schema-version` among substrate-runtime artefacts to never track
- `orbit-state/crates/core/src/layout.rs:12` comment — `schema-version             (substrate-written, gitignored)`

**Recommendation:** Add `.gitignore` at `orbit-state/.orbit/.gitignore` mirroring the toplevel substrate-runtime rules (or just `schema-version` if narrower scope preferred), then `git rm --cached orbit-state/.orbit/schema-version` in a follow-up commit on this branch — or sweep it into the next bumped patch. Not blocking the spec's scope; the file content is harmless (just a version marker that will get rewritten the next time anyone in another clone runs orbit from `orbit-state/`). Flagging because it's the only artefact in the diff that isn't deliberate spec work.
